# 内存优先数据流模式

## 背景

实现书签功能时，先采用了「保存磁盘 → 从磁盘加载 → 更新内存」的数据流。但在磁盘写入静默失败时（如 `_hashPath` RangeError），加载回来的空列表会直接覆盖内存中的正确数据。用户看到"添加成功"但列表始终为空。

## 问题：磁盘优先的隐患

```
用户操作 → 写磁盘（可能失败）→ 读磁盘（拿回失败后的数据）→ setState → UI 空白
```

核心问题：**磁盘 I/O 不可靠**（权限、路径错误、序列化异常），但 UI 反馈依赖磁盘结果。

## 方案：内存优先

```
用户操作 → 立即更新内存（setState）→ 异步写磁盘
        → UI 即时响应 ✅
```

如果写磁盘失败 → catch 异常 → 可选：红色 SnackBar 提示用户
下次进入 → 从磁盘加载 → 如果磁盘写入成功 → 数据一致

## 实现要点

### 1. 增/删操作

```dart
Future<void> _addBookmark() async {
  // 1. 立即更新内存
  _bookmarks = [..._bookmarks, newBookmark];
  _bookmarks.sort(...);
  if (mounted) setState(() {});

  // 2. 异步持久化
  try {
    await Storage.add(newBookmark);
  } catch (e) {
    // 磁盘失败不丢内存数据，仅提示
    showErrorSnackBar(e);
  }
}
```

### 2. 首次加载标记

```dart
bool _bookmarksLoaded = false;

Future<void> _loadFromDisk() async {
  final data = await Storage.load();
  _bookmarksLoaded = true;
  if (mounted) setState(() => _bookmarks = data);
}

void _showList() {
  if (!_bookmarksLoaded) await _loadFromDisk(); // 仅首次
  showSheet(); // 用内存数据
}
```

### 3. 清理（删除最后一条）

```dart
Future<void> _deleteLast() async {
  _bookmarks = [];
  if (mounted) setState(() {});
  
  try {
    await Storage.clear();
  } catch (e) { /* 静默失败，下次进入时磁盘数据可能不一致 */ }
}
```

## 适用场景

| 场景 | 适合？ | 原因 |
|------|--------|------|
| 书签增删 | ✅ | 用户期望即时反馈 |
| 标注增删 | ✅ | 同上 |
| 设置保存 | ❌ | 设置不频繁，磁盘优先可接受 |
| 阅读进度 | ✅ | 边读边存，内存优先防丢 |

## 注意事项

1. **防抖**：频繁操作（如翻页保存进度）应加防抖（300ms Timer）
2. **错误可见**：磁盘失败不应完全静默，至少 `debugPrint`，重要数据用 SnackBar
3. **一致性**：下次进入时磁盘数据覆盖内存（内存数据不保证跨会话）
