# Flutter MVP 技术决策笔记

> 来源：2026-06-03 项目启动讨论

---

## 1. 为什么选 Flutter 而不是其他？

| 方案 | 优势 | 劣势 | MVP 适用性 |
|------|------|------|------------|
| Flutter | 三端一套代码，生态成熟 | Win 端体积偏大（~80MB+） | ✅ 最佳 |
| React Native | 移动端强 | 桌面端弱，依赖社区方案 | ❌ |
| Electron | 文档库丰富（pdf.js 等） | 包体积 150MB+ | ❌ |
| Tauri | 超轻量（~5MB） | Rust 门槛高，移动端无支持 | ❌（移动端不行） |

**关键认知**：三端覆盖是刚需。Tauri 只覆盖桌面端，移动端得另起一套，MVP 阶段投入产出比低。Flutter 是目前唯一能**一套代码三端**的成熟方案。

**兜底策略**：如果 Windows 端体积成为痛点，后期可将 Windows 端迁移到 Tauri，保留 Flutter 做移动端。

## 2. 标注 vs 编辑：阅读理解器的边界

**核心原则**：阅读器 ≠ 编辑器。

- 标注数据**独立存储**，不侵入原文 —— 类似 PDF 的 annotation layer
- 好处：
  - 保护版权（不修改原文件）
  - 标注可单独同步/导出
  - 保持轻量
- 真要编辑内容 → 用 Typora / Word / Adobe

**技术含义**：
- 标注是独立的数据模型，和文档解析解耦
- 渲染时标注层叠加在文档之上
- 标注数据格式可能是 JSON（便于跨平台同步）

## 3. Markdown 作为格式转换中枢

**思路**：各格式 → MD → 各格式。

- MD 是纯文本，解析简单
- MD 可以承载富文本语义（标题、列表、加粗、链接、表格）
- 已有大量 MD → 其他格式的转换器（Pandoc 等）
- 市场上"万物皆可 MD"的趋势验证了这个方向

**技术选型考量**：
- PDF → MD：需要 OCR 或文本提取库
- EPUB → MD：EPUB 本质是 HTML+CSS，可提取内容转 MD
- MD → PDF：Flutter 侧渲染或调用转换库

---

## 4. 格式解析难度分级

| 格式 | 难度 | 原因 |
|------|------|------|
| TXT | ⭐ | 纯文本，只需处理编码 |
| MD | ⭐ | 有成熟 Dart 库（flutter_markdown） |
| 图片 | ⭐ | Flutter 原生支持 |
| EPUB | ⭐⭐ | 本质是 ZIP 包 + HTML，解析不难，排版需处理回流 |
| PDF | ⭐⭐⭐ | 固定版面，文字提取和渲染都需要专门库 |
| DOCX | ⭐⭐⭐ | OOXML 格式，解析复杂 |
| MOBI | ⭐⭐⭐ | 二进制格式，文档少 |
| AZW/AZW3 | ⭐⭐⭐⭐ | Amazon 专有 + DRM |

**MVP 策略**：先攻 PDF 和 EPUB —— 这两个是最核心的阅读格式。
