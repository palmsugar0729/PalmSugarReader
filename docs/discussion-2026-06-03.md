## 一、技术栈决策

**问题**：三端（Windows / Android / iOS）用什么技术栈？

**讨论**：
- Flutter：一套代码三端跑，轻量，生态可用
- React Native：移动端强，桌面端弱
- Electron：桌面包体积大（Chromium 百兆起步）
- Tauri：轻量但 Rust 学习曲线陡

**结论**：**Flutter MVP 先行**。功能稳定后，若 Windows 端 Flutter 打包体积过大，再考虑迁移到 Tauri + Flutter 混合方案。

---

## 二、编辑功能范围

**问题**："简单的编辑功能"到底多简单？

**讨论**：
- 用户痛点是现有 PDF 阅读器（极光 PDF）太臃肿、收费
- 真正的文档编辑应在专业编辑器（Typora、Word）中完成
- 标注（高亮、划线、批注）已足够覆盖阅读场景
- 只做标注也有利于版权保护

**结论**：**编辑 = 标注层**，不修改原文内容。

---

## 三、格式优先级

**问题**：支持哪些格式？先做哪些？

**结论**：
- P0 优先：PDF、EPUB、TXT、MD、图片
- P1 随后：MOBI、DOCX
- P2 后期：AZW/AZW3、DOC

---

## 四、格式转换

**用户需求**：增加格式转换功能。市场上出现了"万物皆可 MD"的转换器，说明 Markdown 作为中间格式的路线可行。

**结论**：以 MD 为核心中转格式，实现常见格式互转。
